# 第 7 章：从原型到生产——理解差距

> **段位目标：L4** | ⏱️ 30 分钟 | 对照 Claude Code 官方源码

这一章不需要深入读代码。只需要理解：**从 1300 行到 51 万行，多出来的是什么产品能力。**

## 7.1 差距全景

| 产品能力 | 教学版 (1300行) | 官方版 (51万行) | 差距倍数 |
|---|---|---|---|
| 工具数量 | 6 个 | 42+ 个 | 7x |
| 工具发现 | 全部暴露 | 延迟加载 + 搜索发现 | — |
| 上下文管理 | 1 层压缩 | 5 层分级压缩 | 5x |
| 安全机制 | 正则匹配 + 确认 | 5 层权限体系 | 5x |
| UI | console.log | React + Ink 组件化 | — |
| IDE 集成 | 无 | VS Code + JetBrains | — |
| 多智能体 | 单 Agent | Agent + Team + Coordinator | — |
| 插件/技能 | 无 | 完整生态系统 | — |
| 启动优化 | 即时 | 并行预取 + 延迟加载 | — |
| API 后端 | Anthropic + OpenAI | 仅 Anthropic | — |

## 7.2 三个产品层面最重要的认知

### ① 工具发现机制是规模化的关键

- 6 个工具可以全部塞进 prompt
- 42 个不行——模型会选错
- 官方的 `ToolSearchTool`：模型先搜索「我需要哪种工具」，再加载

> 🎯 **产品规则：当工具超过 15 个时，必须引入工具发现/分组机制。**

### ② 多智能体是任务分解策略

官方的 `AgentTool` 可以生成子 Agent：

```
主 Agent → 生成子 Agent（负责搜索相关文件）
        → 生成子 Agent（负责修改代码）
        → 生成子 Agent（负责运行测试）
```

每个子 Agent 有独立的上下文窗口和受限的工具集。

> 🎯 **产品含义：** 复杂任务（如跨 10 个文件的重构）可以拆成多个小任务并行执行。一个 200K 上下文窗口如果不够，三个子 Agent 就是 3 × 200K。

### ③ IDE 集成 = 把 AI 嵌入已有工作流

官方的 Bridge 系统（31 个文件）就是为了让用户**不离开 VS Code** 就能用 Claude Code。

> 🎯 **产品策略：** 不要让用户来你的产品，把你的产品带到用户已有的使用场景里。考虑做浏览器插件、Slack bot、IDE 扩展，而不是独立 App。

## 7.3 功能标志系统

官方版用 20+ 个 feature flags 控制功能：

| 标志 | 功能 | 产品含义 |
|---|---|---|
| `VOICE_MODE` | 语音输入 | 新交互方式灰度发布 |
| `COORDINATOR_MODE` | 多智能体协调 | 高级功能分阶段开放 |
| `CONTEXT_COLLAPSE` | 上下文折叠 | 性能优化 A/B 测试 |
| `MCP_SKILLS` | MCP 技能 | 平台能力渐进开放 |

**产品启示：** 复杂 Agent 产品需要 feature flag 系统做灰度发布。不要一次全量上线。

## 7.4 从教学版你能学到什么，不能学到什么

### ✅ 能学到的（核心原理）
- Agent 循环的本质
- 工具设计与描述
- 提示词架构
- 成本模型
- 基本安全设计

### ❌ 不能学到的（需要实际经验）
- 大规模工具管理
- 多智能体协调策略
- 企业级安全合规
- 性能优化（启动时间、并发）
- 平台化（插件、技能生态）

**这没关系。** 核心原理掌握后，生产化的问题可以在实际产品中逐步解决。
